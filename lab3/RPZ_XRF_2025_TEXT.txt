На основе анализа структурных данных и предполагаемой нагрузки был произведен расчет аппаратных
требований для системы заявок на XRF-анализ. Система рассчитана на одновременную работу до 100
создателей и 10 модераторов. Пусть один создатель формирует 30 заявок в день, рабочий день 8 часов.
Количество заявок в сутки определяется по формуле 1.

N_claims_day = N_creators * claims_per_creator_day                           (1)
N_claims_day = 100 * 30 = 3000 заявок/сутки

На каждую заявку по диаграмме последовательности приходится в среднем 16 запросов к серверу.
Дополнительно модераторы выполняют polling 1 запрос/с. RPS вычисляется по формуле 2.

RPS = (N_claims_day * requests_per_claim) / (work_hours * 3600) + N_moderators * polling_rps   (2)
RPS = (3000 * 16) / (8 * 3600) + 10 * 1 = 11.67 rps

Рассчитаем количество ядер по формуле 3, примем kreserve = 1.5, а необходимый объем ОЗУ по формуле 4
с учетом того, что на одно ядро требуется 4 ГБ оперативной памяти.

CPU_cores = ceil((RPS / RPS_per_core) * kreserve), где RPS_per_core = 5      (3)
CPU_cores = ceil((11.67 / 5) * 1.5) = 4 ядра

RAM_GB = CPU_cores * 4                                                        (4)
RAM_GB = 4 * 4 = 16 ГБ

Рассчитаем размеры записей в таблицах artifact_claims и claim_alloy_matches по формулам 5 и 6.
Для artifact_claims примем средний размер 650 байт/запись (включая служебный overhead).
Для claim_alloy_matches примем 48 байт/запись. Среднее число услуг в заявке примем равным 3.

Row_claim = 650 байт                                                          (5)
Row_match = 48 байт                                                           (6)

Рассчитаем годовой объем данных и необходимый объем диска.

Data_day = N_claims_day * Row_claim + N_claims_day * 3 * Row_match           (7)
Data_day = 3000 * 650 + 3000 * 3 * 48 = 2 382 000 байт/сутки

Data_year = Data_day * 365                                                    (8)
Data_year ≈ 869 430 000 байт ≈ 0.81 ГБ/год

Disk_required = Data_year + Nreserve, где Nreserve = 2 ГБ                    (9)
Disk_required ≈ 2.81 ГБ

Таким образом, для серверного окружения системы достаточно конфигурации: 4 vCPU, 16 ГБ RAM, SSD от 20 ГБ
с учетом резерва на рост данных, журналов и резервных копий.

Москва - 2025

Цель
Необходимо разработать систему, включающую веб-сервис, SPA-приложение и нативное приложение,
которая позволит пользователям формировать и отправлять заявки на XRF-анализ артефактов.

Назначение разработки
Необходимо разработать систему, которая автоматизирует работу с заявками на XRF-анализ и
предоставляет актуальную информацию об эталонных услугах. Создатели смогут формировать заявки,
добавлять услуги, редактировать параметры анализа и формировать заявку для модерации. Модераторы
смогут завершать либо отклонять сформированные заявки. Гости системы имеют доступ к просмотру
списка услуг и карточек услуг, а также к регистрации и аутентификации.

Стадии и этапы разработки
Создание дизайна приложения в Figma, развертывание MinIO
Создание базы данных для хранения информации об услугах, пользователях и заявках (PostgreSQL, GORM)
Создание веб-сервиса со всей бизнес-логикой на Golang
Добавление JWT-авторизации, Redis-сессий и Swagger в веб-сервис
Разработка SPA на React для гостя
Внедрение адаптивности, добавление менеджера состояний Redux Toolkit
Разработка интерфейса создателя/модератора на React
Внедрение адаптивности и развертывание приложения
Реализация нативного приложения на Tauri
Развертывание веб-приложения на GitHub Pages
Подготовка набора документации (РПЗ, ТЗ, набор диаграмм)
Оформление git-репозитория на сервисе GitHub

Требования к авторизации и правам
Без авторизации доступны только операции чтения услуг и методы регистрации/аутентификации.
С авторизацией для роли creator доступны пользовательские методы заявок.
Для роли moderator доступны все методы creator и метод модерации заявок.
Для гостя методы заявок возвращают 401.
Для creator метод /api/claims/:id/moderate возвращает 403.

Меню
Главная – перенаправляет на страницу списка услуг
Список услуг – вызывает метод GET /api/services
Список заявок – вызывает метод GET /api/claims (для авторизованных)
Корзина – вызывает метод GET /api/claims/cart-icon
Регистрация – вызывает метод POST /api/users/register
Личный кабинет – показывает текущего пользователя из JWT claims
